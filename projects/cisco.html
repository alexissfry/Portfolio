Overview

During Summer 2023, I worked at Cisco on the Cloud Platform team within the Security Business Group (SBG). My team works with all product, infrastructure, and data teams within Cisco Secure to provide and support common engineering tools.

Problem

During my time at Cisco, my team was doing a large-scale migration for all teams from BitBucket to GitHub. My project was an end-to-end automation platform that would allow internal engineers to post a ticket on our kanban board and automatically get added to the SBG GitHub organization, replacing an otherwise manual process.

Process

Design Decisions

Before jumping into implementation, I met with my team to discuss the underlying design of the system. A very important decision was if the automation was going to be built with an API gateway or a Flask web app, so we weighed the most relevant pros and cons.

Ultimately, we decided for the foundation of the automation platform to be a Flask web app due to its accessible, extensible nature as it will change to automate more tasks, my prior experience using this framework, and ability to scale.

Technical Architecture

This diagram shows the flow of information through my application. The workflow starts by a user posting a ticket to our Jira board, specifying a task ‚Äúcode‚Äù in the title, and giving the Cisco ID of the user that needed to be added in the description. Upon the creation of a ticket, a webhook would send the ticket data in the form of a JSON to my application, which would then need to be parsed to trigger calls to certain endpoints and, in turn, interface with the Jira, GitHib, and Cisco MyID REST APIs. A result wouldis then be returned back to the user on the original Jira ticket, stating whether they were successfully onboarded onto GitHub or not.

Implementation: Balancing UX & Dev Time

One technical takeaway from this project was a design decision I needed to make when figuring out how to parse data. During testing, I noticed that if a user applied any styling, such as a hyperlink or italics, to within the body of the ticket request, the JSON shape would change and my code would be unable to find the user ID in the expected layer of the description. I had two options: to tell the user not to apply styling or links to their ticket text or to rebuild and test a new parsing class. I realized that adding this specification to the automation platform‚Äôs user docs could be easily missed and cause lots of frustration if a user‚Äôs request fails. Thus, I opted to put the user first and redevelop the parsing component of the application.

Throughout the development process, I wrote units in pytest to ensure that the endpoints of each function were working as expected leveraging mocked ticket data. Thorough testing was especially important for this project as it was necessary to parse commands from Jira tickets which might not properly follow the code schema I created. In both success cases and a variety of error cases, such as incorrectly formatted tickets and system errors, it was necessary to send different communications back to the user.

I also used Postman to intercept and send test API requests to my application to make sure that the success and error responses were returning as expected. This was especially useful as the automation platform interacted with multiple APIs, and I could ensure that the platform was sending the proper requests to such interfaces.
